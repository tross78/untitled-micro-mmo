# P2P Micro-MMO Architecture Plan

## Background & Motivation
The goal is to build a highly scalable (up to 1M concurrent users) micro-MMO with a near-zero infrastructure budget (<$5/month). The system will rely on decentralized technologies orchestrated by **Trystero**, and a "botnet-style" architecture using trusted headless nodes (like Raspberry Pis) for state persistence and global events, rather than traditional centralized servers.

## Core Gameplay & Scope
- **Bootstrapping (MVP):** The game will initially launch as a **Text-Based MUD** (Multi-User Dungeon). This minimizes asset size, perfectly aligns with the bandwidth constraints of the Pi, and matches the slow latency tolerance of the P2P networking model. 
- **Future Expansion:** The architecture is designed to support a graphical client (e.g., using Kontra.js or WASM) in later phases once the core state and networking layers are stable.
- **Global Events:** Inspired by "Blaseball", the game features overarching events that impact all concurrent active rooms. These are procedurally generated by an ultra-lightweight LLM running directly on the trusted headless node.
- **Distribution:** The initial HTML/JS/CSS client is extremely small and distributed via GitHub Pages.

## Network Architecture & Topology

### 1. The "Command & Control" (C&C) Headless Nodes
Instead of a traditional authoritative server, the system uses a network of trusted, headless clients (starting with a single Raspberry Pi Zero W).
- **Role:** These nodes idle in the network, persisting the master state of various rooms, handling conflict resolution (via Yjs CRDTs), and generating/broadcasting global narrative events.
- **Authority:** They hold the master cryptographic keys needed to sign official state changes, preventing malicious peers from permanently corrupting the world.

### 2. Trackerless Signaling via Trystero
To maintain strict $0 infrastructure costs for signaling and eliminate the complexity of raw WebRTC:
- **Trystero Integration:** We will use the Trystero library (BitTorrent strategy) to completely abstract WebRTC matchmaking. It automatically uses existing public infrastructure to exchange SDP offers and establish P2P connections.
- **Mechanism:** Players join a conceptual "room" (representing a map instance) using `trystero.joinRoom()`. Trystero handles the NAT traversal, STUN configuration, and underlying peer-to-peer data channels automatically.

### 3. Room-Based Mesh Topology
- The world is sharded into discrete instances/rooms.
- Players form small WebRTC mesh networks (e.g., max 10-20 peers) per room managed by Trystero.
- State changes (moves, combat actions) are broadcasted within the room's mesh using Yjs over Trystero's data channels to ensure eventual consistency.
- A headless node acts as a "lurker" in populated rooms to persist the final state when the room empties.

## Technical Stack
- **Client UI (Phase 1):** Vanilla JS / HTML / CSS (Text-based DOM manipulation).
- **Client UI (Phase 5+):** Kontra.js (or similar ultra-lightweight engine/WASM binary).
- **Networking:** Trystero (@trystero-p2p/torrent).
- **State Management:** Yjs (Lightweight CRDT).
- **Cryptography:** Ed25519 signatures (via tweetnacl for verifying player actions and arbiter events).
- **Story Generation:** `llama.zero` (ARMv6 optimized fork of `llama.cpp`) running an ultra-small quantized model (e.g., `SmolLM2-135M-Instruct-Q4_K_M`).

## Implementation Phases

### Phase 1: The Text-Based Bootstrap Layer (COMPLETED)
1. Set up the GitHub Pages repository for the initial HTML/JS shell (purely text-based UI).
2. Establish the "Blaseball" aesthetic (no native chat, entirely action/event driven).
3. Achieve a baseline bootstrap JS bundle size under 175KB (currently ~154KB).

### Phase 2: Trystero Networking & Mesh Setup (COMPLETED)
1. Implement Trystero's `joinRoom` API to group players into specific game instances (shards).
2. Wire up Trystero's `makeAction` handlers to broadcast and receive player text commands (`/move`).
3. Test peer connection stability, latency, and NAT traversal success rates.

### Phase 3: State Management & The Headless Pi (COMPLETED)
1. Integrate Yjs for shared state (world ticker, player maps) within a room's mesh network.
2. Develop the Node.js headless client (`arbiter/index.js`) for the Raspberry Pi using `werift`.
3. Implement the C&C logic: The Pi uses Trystero to join active rooms, signs global narrative events, and broadcasts them to clients for verification.

### Phase 4: Security Hardening & Micro-LLM Events
1. Implement the strict deterministic ruleset on the Pi to validate all Yjs state changes (see Security below).
2. Compile `llama.zero` on the Raspberry Pi Zero W (requires 2GB swap file).
3. Implement the slow event loop: The Pi generates narrative events (0.1 - 0.5 tok/s), signs them, and broadcasts them as global "Blaseball" events.

### Phase 5: Graphical Client (Future)
1. Once the MUD is stable, introduce Kontra.js to render the text state into a 2D graphical representation.

## Security & Anti-Cheat (Crucial for P2P)
Because clients run locally and connect directly to each other, the network is inherently untrusted. "Hackers" will try to send modified data to peers.
- **End-to-End Encryption:** Trystero utilizes standard WebRTC DTLS/SRTP, meaning ISPs or middlemen cannot read or inject packets.
- **Action Signatures:** Every action (e.g., "Attack Goblin", "Move North") is signed by the player's unique Ed25519 private key generated on their first visit.
- **Deterministic State Validation:** Every client runs the same logic.
  - When peer A sends an action to peer B, peer B runs the logic locally to verify the outcome is legal before merging it into their CRDT.
- **The Arbiter Node (The Pi):** The trusted headless node constantly lurks in active rooms. If it detects an impossible state change (e.g., a player moving through a wall or giving themselves infinite items), the Pi broadcasts an authoritative rollback signed by the Master Key, and permanently blacklists the offending player's public key across all instances.

## Constraints & Mitigation
- **Pi Zero W Limitations:** Running an LLM on 512MB RAM/ARMv6 requires a specialized fork (`llama.zero`) and limits output to ~0.5 tokens/sec. This extreme slowness is mitigated by the asynchronous nature of the "Blaseball" event loop.
- **Signaling Reliability:** Public BitTorrent/Nostr relays used by Trystero can sometimes be slow. The text-based UI will gracefully display "Dialing network..." to manage expectations.

## Migration & Rollback
Updates to the core game engine or ruleset are handled by uploading a new script and updating the GitHub Pages static bootstrap. If an update breaks the game, reverting the commit on GitHub Pages instantly points new clients back to the old, stable version.
