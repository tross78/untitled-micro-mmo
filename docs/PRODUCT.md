# Product

## Elevator Pitch

Fenhollow is a mobile-first browser MMO slice where most play is solo, but everyone inhabits the same Arbiter-synced world. The target feel is closer to a compact shared-world life/adventure game than a feature-complete social MMO.

Short version:

* "Blaseball and Stardew Valley had a baby"
* SNES-Zelda-grade readability
* small shared world, weird daily shifts, low-friction sessions

## Audience and Launch Shape

* Start small with friends and a trusted early-feedback group
* Mobile first, desktop second
* Browser only
* Monetization is not part of the near-term roadmap

## Product Weighting

* 75% solo
* 25% multiplayer

The multiplayer value is not constant chat volume. The differentiator is that all players feel the same world's shifts, shortages, danger spikes, and daily texture.

## Current First-Class Loop

The intended loop is:

* move through town and nearby danger zones
* fight, loot, craft, recover
* use shops, bank, mill, and herbalist as support economy nodes
* complete quests and push into slightly riskier spaces
* return tomorrow to a world that feels a little different

## What Is Not Core Right Now

These are distractions unless explicitly revived:

* chat-style `say`
* emote commands
* `vision`
* first-class trading UX
* first-class dueling UX

## Product Priorities

Near-term priority order:

1. gameplay feel
2. retention loop
3. launch readiness

## Arbiter Role in the Product

The Arbiter is not just infrastructure. It should become a visible product feature:

* synchronized daily world changes
* scarcity/surplus shifts
* danger shifts
* shared events people notice and talk about

If a feature ignores that shared-world premise, it is probably not top priority.
